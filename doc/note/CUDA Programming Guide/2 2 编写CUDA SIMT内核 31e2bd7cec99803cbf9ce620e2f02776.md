# 2.2 编写CUDA SIMT内核

介绍GPU 上线程的**调度机制**、**内存访问**方式以及**执行流程**。

## **SIMT 基础概念**

CUDA 线程是并行执行的基本单位。线程束与 SIMT 架构阐述了 GPU 执行的基本 SIMT 模型。

SIMT 模型允许每个线程保持独立状态和控制流。但需要注意让内核代码尽量减少同一线程束中线程执行 divergent（分化）代码路径的情况

## 线程层次结构 Thread Hierarchy

线程被组织为线程块，线程块进而组织成网格。

- 网格可以是一维、二维或三维的，网格的大小可以在内核中通过内置变量 `gridDim` 查询。
- 线程块同样可以是一维、二维或三维的。线程块的大小可以在内核中通过内置变量 `blockDim` 查询。
- 线程块的索引可通过内置变量 `blockIdx` 查询。
- 在线程块内部，线程的索引通过内置变量 `threadIdx` 获取。

这些内置变量用于计算每个线程**唯一的全局线程索引，**从而使每个线程能够根据需求从全局内存加载/存储特定数据并**执行独立的代码路径**。

<aside>
💡

- `gridDim.[x|y|z]`：分别表示网格在 `x` 、 `y` 和 `z` 维度上的大小。这些数值在内核启动时设定。
- `blockDim.[x|y|z]` ：分别表示线程块在 `x` 、 `y` 和 `z` 维度上的大小。这些数值在内核启动时设定。
- `blockIdx.[x|y|z]` ：分别在 `x` 、 `y` 和 `z` 维度上的块索引。这些值根据当前执行的块而变化。
- `threadIdx.[x|y|z]` ：分别在 `x` 、 `y` 和 `z` 维度上的线程索引。这些值根据当前执行的线程而变化。
</aside>

线程块内的线程会按可预测的顺序进行线性化：**x 是最内层维度（变化最快），其次是 y，z 是最外层维度（变化最慢）**。计算代码如下：

```cpp
// 线程在块内的索引
thread_idx_in_block = threadIdx.z * BlockDim.y * BlockDim.x + 
											threadIdx.y * BlockDim.x + 
											threadIdx.x
```

连续的 threadIdx.x 值对应连续的线程；threadIdx.y 每增加 1，线性化索引增加 blockDim.x；threadIdx.z 每增加 1，线性化索引增加 blockDim.x * blockDim.y。

**块在网格的索引为**：

```cpp

// 块在网格内的索引 
block_idx_in_grid = blockIdx.z * gridDim.y * gridDim.x +
										blockIdx.y * gridDim.x + 
										blockIdx.x
```

线程在网格中的全局索引为：

```cpp
// 线程在网格中的全局索引
thread_idx_in_grid = block_idx_in_grid * (blockDim.z * blockDim.y * blockDim.x) + 
										 thread_idx_in_block
```

### 全局内存 Global Memory

全局内存（亦称设备内存）是用于存储数据的主要内存空间，可供**核函数中的所有线程访问**。

全局内存具有**持久性【表格中提到他的生命周期是Application，Location在设备上】**。这意味着全局内存中的分配操作及其存储的数据将一直保留，直到该内存被释放或应用程序终止。 `cudaDeviceReset` 会释放所有已分配的内存。

- 全局内存通过 CUDA API 调用（如 `cudaMalloc` 和 `cudaMallocManaged` ）进行分配。
- 数据可通过 CUDA 运行时 API 调用（例如 `cudaMemcpy` ）从 CPU 内存复制到全局内存。
- 使用 CUDA API 分配的全局内存需通过 `cudaFree` 释放。

在内核启动前，全局内存由 CUDA API 调用完成分配与初始化。

内核执行期间，CUDA 线程可从全局内存读取数据，并将运算结果写回全局内存。**【由于全局内存可被网格中所有线程访问，必须谨慎避免线程间的数据竞争（data races）。】**

当内核执行完毕后，其写入全局内存的结果既可复制回主机，也可供 GPU 上其他内核使用。**【通过由于CUDA内核的返回类型为`void` ，内核计算的数值结果若要返回host则必须写入全局内存。】**

```cpp
__global__ void VecAdd(float* A, float* B, float* C, int vectorLength)
{
		int workIndex = threadIdx.x + BlockIdx.x * BlockDim.x;
		if(workIndex < vectoLength)
		{
				C[workIndex] = A[workIndex] + B[workIndex];
		}
	
}
```

<aside>
💡

代码展示的是位于全局内存的三个数组A、B、C被向量加法内核访问

</aside>

### 共享内存 Shared Memory

共享内存是一种可供**线程块内所有线程**访问的内存空间。**【因为它物理上位于SM中，同时因为是Block内所有线程可访问，应避免数据竞争】**与 **L1 缓存**、**统一数据缓存**使用相同的物理资源。

共享内存中的数据在**整个内核执行期间**持续存在。**【指的是线程块在执行内核期间】**

共享内存可视为用户管理的临时缓存，容量相较于全局内存较小，但由于其位于每个 SM 上，其访问**带宽更高、延迟更低**。

同一线程块内线程间的同步可通过 `__syncthreads()` 函数实现。此函数会**阻塞线程块内的所有线程**，直至所有线程都执行到对 `__syncthreads()` 的调用。

```cpp
// Assuming blockDim.x is 128
__global__ void example_syncthreads(int* input_data, int* output_data)
{
		__shared__ int shared_data[128];
		// Every thread writes to a dinstinct element of 'shared data':
		shared_data[threadIdx.x] = input_data[threadIdx.x];
		
		// All threads synchronize, guaranteeing all writes to 'shared data' are ordered
		// before any thread is unblocked from '__syncthreads()':
		__syncthreads();
		
		// A single thread safely reads 'shared_data':
		if (threadIdx.x == 0)
		{
				int sum = 0;
				for(int i = 0; i < BlockDim.x; i++)
				{
						sum += shared_data[i];
				}
				output_data[blockIdx.x] = sum;
		}
}
```

<aside>
💡

判断 threadIdx.x == 0 的核心原因是：只让**一个线程执行**“汇总计算和写回”操作，**避免重复计算和写冲突**。

`__syncthreads()`和`cudaDeviceSynchronize()`的区别：

- `__syncthreads()`作用范围是**同一线程块内的所有线程，位置在kernel内部。**目的是协调块内线程，例如Shared Memory协作。
- `cudaDeviceSynchronize()`作用范围在**整个GPU上的所有kernel**，调用位置是CPU，目的是**等待GPU执行任务完成**。
</aside>

由于**共享内存和 L1 缓存共享同一物理空间**，使用共享内存会缩减内核可用的 L1 缓存容量。若内核未使用共享内存，则全部物理空间将被 L1 缓存占用。

- 通过 `cudaGetDeviceProperties` 函数查询每个流式多处理器（SM）及每个线程块的共享内存大小。
- 或探查 `cudaDeviceProp.sharedMemPerMultiprocessor` 与`cudaDeviceProp.sharedMemPerBlock` 设备属性获得。
- CUDA  Runtime API 提供了函数 `cudaFuncSetCacheConfig` ，用于告知运行时应当为共享内存分配更多空间，还是为 L1 缓存分配更多空间。该函数向运行时**传递的是偏好设置**，但运行时不保证完全遵循此设置。

共享内存既可按**静态方式分配**，也可按**动态方式分配**。

- **共享内存的静态分配 Static Allocation of Shared Memory**
    
    静态共享内存分配必须在**内核中**使用`__shared__`说明符声明变量。以这种方式声明的**共享内存大小**必须在**编译时确定**。
    
    ```cpp
    __shared__ float sharedArray[1024]; 
    
    // Usually using __syncthreads()
    ```
    
- **共享内存的动态分配 Dynamic Allocation of Shared Memory**
    
    内核启动的**三重尖括号表示法中指定每个线程块所需的共享内存大小（以字节为单位）**，将其作为第三个可选参数`functionName<<<grid, block, sharedMemoryBytes>>>()`.
    
    随后，在内核内部，程序员可以使用 `extern __shared__` 限定符声明一个变量，该变量将在**内核启动时进行动态分配**。
    
    ```cpp
    extern __shared__ float sharedArray[];
    ```
    
    若希望创建**多个**动态分配的共享内存数组，则必须使用**指针运算手动分割**`extern __shared__`的地址。例如：
    
    ```cpp
    /* target
    short array0[128];
    float array1[64];
    int array2[256];
    */
    
    extern __shared__ float sharedArray[];
    
    // array0_address_start = base
    short* array0 = (short*)array;
    // array1_address_start = base + 128 * sizeif(short)
    float* array1 = (float*)&array0[128];
    // array2_address_start = base + 128 * sizeof(short) + 64 * sizeof(float)
    int* array2 = (int*)&array1[64];
    ```
    
    指针需要与其指向的数据类型**对齐**，因此例如以下代码无法正常工作，因为 `array1` 未按 **4 字节对齐**。地址计算说明：
    
    offset = 127 * sizeof(short) = 254，address = base + 254;
    
    254 mod sizeof(float) = 2, address mod sizeof(float) ≠ 0，本质原因是地址无法满足对齐要求。
    
    ```cpp
    extern __shared__ float sharedArray[];
    
    short* array0 = (short*)array;
    float* array1 = (float*)array0[127];
    ```
    

### 寄存器 Registers

寄存器位于**流多处理器**上，具有**线程局部作用域**。寄存器的使用**由编译器管理**，在内核执行期间，寄存器用于**线程的本地存储**。

每个SM上的寄存器数量以及每个线程块的寄存器数量可以通过 GPU 的  `regsPerMultiprocessor` 和  `regsPerBlock` 设备属性进行查询。

NVCC 允许开发者通过 `-maxrregcount` 选项指定内核使用的最大寄存器数量。

### 本地内存 Local Memory

本地内存是一种**类似于寄存器**的线程局部存储空间，由 NVCC 管理，但其物理位置位于**全局内存空间（“本地”仅指它的逻辑作用域范围）**中。**【与寄存器不同的是它的位置在Device上】**

编译器可能将以下自动变量放入本地内存：

- **无法确定其索引**是否使用常量值的数组，
- **占用过多寄存器空间**的大型结构体或数组。
- 任何变量，如果内核使用的寄存器数量超出可用范围，即发生**寄存器溢出**。

由于本地内存空间位于设备内存中，本地内存访问具有与全局内存访问相同的延迟和带宽。并且需要满足与《合并全局内存访问》章节所述相同的**内存合并要求。【理解：本地内存需要满足内存合并要求（Memory Coalescing），合并访问才能达到全局内存访问的延迟和带宽。】**

### 常量内存 Constant Mmemory

常量内存具有网格作用域，在应用程序生命周期内可访问。常量内存位于设备上且**对内核为只读状态**。必须在**主机端**使用 `__constant__` 限定符在**任何函数之外**进行**声明和初始化**。

`__constant__` 内存空间限定符所声明的变量具有以下特性：

- 驻留在**常量内存空间**，
- 其**生命周期**与创建它的 **CUDA 上下文相同**，
- **每个设备**对应**独立**的对象，
- **网格内所有线程**及通过Runtime库的**主机**均可访问（`cudaGetSymbolAddress()` / `cudaGetSymbolSize()` / `cudaMemcpyToSymbol()` / `cudaMemcpyFromSymbol()` ）。

可通过**查询** `totalConstMem` 设备属性元素获取常量内存的总容量。相较于其他类型内存，常量内存适用于少量数据，容量较小，通常为**每设备 64KB**。

以下代码段为常量内存的声明与使用方法：

```cpp
// in your .cu file
__constant__ float coeffs[4];

__global__ void compute(float* out)
{
		int idx = threadIdx.x;
		out[idx] = coeffs[0] * idx + coeffs[1];
}

// In your host code
float h_coeffs[4] = {1.0f, 2.0f, 3.0f, 4.0f};
cudaMemcpyToSymbol(coeffs, h_coeffs, sizeof(h_coeffs));
compute<<<1, 10>>>(device_out);
```

### 缓存 Caches

GPU 设备采用**多级缓存结构**，包括 **L2** 与 **L1 缓存**。

- L2 缓存位于**设备端**，由**所有流多处理器（SM）共享**。可通过函数 `cudaGetDeviceProperties` 中的设备属性元素 `l2CacheSize` 查询 L2 缓存的大小。
- L1 缓存在物理上位于每个流多处理器（SM）内部，并与共享内存使用相同的物理存储空间。

L2 和 L1 缓存可通过**特定函数进行控制**，使开发者能够指定多种缓存行为。

### **纹理与表面内存 Texture and Surface Memory**

用于**将图像数据加载为 3D 渲染中的纹理**。CUDA 通过**纹理对象 API 和表面对象 API** 公开了这些指令及其使用机制。

<aside>
💡

部分**旧版 CUDA 代码可能使用纹理内存**，因为在早期的 NVIDIA GPU 中，这种做法在某些场景下能带来性能优势。在当前所有受支持的 GPU 上，这些场景均可通过直接加载与存储指令处理，使用纹理与表面内存指令已无法获得任何性能提升。

</aside>

### 分布式共享内存 Distributed Shared Memory

在**计算能力 9.0** 中引入(RTX50系以上)，并通过**协作组（Cooperative Groups）**实现的**线程块集群**，使得集群内的线程能够访问**该集群中所有参与线程块**的共享内存。**【理解：一般共享内存的作用范围仅在同一线程块内的所有线程，通过协作组可以访问集群中参与线程块的共享内存】**

这种内存共享方式被称为**分布式共享内存**，对应的地址空间则称为**分布式共享内存地址空间**。

属于线程块集群的**线程**可以在**分布式地址空间中进行读取、写入或原子操作**，**无论**该地址属于**本地线程块**还是**远程线程块**。

分布式共享内存的**大小**即为每个集群的**线程块数量**乘以**每个线程块**的**共享内存大小**。

## 内存性能 Memory Performance

确保**合理使用内存**是实现 CUDA 内核高性能的关键。本节将讨论在 CUDA 内核中实现**高内存吞吐量**的一些**通用原则与示例**。

### 合并全局内存访问 Coalesced Global Memory Access

全局存储器通过 32 字节的**内存事务(memory transactions)【意思是GPU 对内存进行的一次完整访问操作】**进行访问。

当 CUDA 线程从全局存储器请求**一个数据字（Word）**时，相关**线程束**会**根据每个线程访问**的**数据字大小**以及**内存地址在线程间的分布情况**，将该线程束中**所有线程的内存请求合并**为满足该请求所需的内存事务数量。

<aside>
💡

例子：若线程请求一个4字节的数据字，线程束向全局存储器发起的内存事务总量将为32字节。**【理解：它的单位是32字节，每次请求都会发回连续32字节的事务，因此只请求4字节的数据字会造成28字节的带宽浪费。】**

</aside>

为最高效地利用内存系统，**线程束**应**充分利用单次内存事务**中提取的**所有数据**。

<aside>
💡

例子：如果某个线程从全局储存器请求4字节的数据字，而事务大小为32字节时，如果**当前线程束有其他线程**，能够使用该32字节的**其他4字节**的数据字，这将实现内存系统的最高效利用

</aside>

![image.png](2%202%20%E7%BC%96%E5%86%99CUDA%20SIMT%E5%86%85%E6%A0%B8/image.png)

上图展示的是当前线程束中的**连续线程**请求内存中连续的 4 字节字，则该线程束总共将请求 128 字节的内存，而这所需的 128 字节将通过四次 32 字节内存事务来获取。这实现了内存系统 100%的利用率。**【理解：连续线程访问连续地址，transaction之间是连续的，请求落在连续的4个事务中（GPU控制器会向下对齐到transaction边界），所有能得到连续块】**

最糟糕的情况是连续线程访问内存中相隔 32 字节或更远的数据元素。**【理解：这种情况下，线程束会向全局储存器为每个线程发出32字节的内存事务，将会获得连续的32字节*32线程/线程束=1024字节的内存流量。利用率为128字节/1024字节=12.5%】**

![image.png](2%202%20%E7%BC%96%E5%86%99CUDA%20SIMT%E5%86%85%E6%A0%B8/image%201.png)

---

实现合并内存访问最直接的方法是让**连续的线程访问内存中连续的元素**。例如：

```cpp
__global__ void vecAdd(float* A, float* B, float* C, int vectorLength)
{
    int workIndex = threadIdx.x + blockIdx.x*blockDim.x;
    if(workIndex < vectorLength)
    {
        C[workIndex] = A[workIndex] + B[workIndex];
```

<aside>
💡

理解：若尾部线程束少于32，线程无法覆盖transaction的边界，那么最后一个事务内部数据会超出实际需要，造成利用率下降。

</aside>

实现合并内存访问**并不要求**连续线程访问连续的内存元素，这只是实现合并的典型方式。

只要一个线程束中的所有线程以**某种线性或置换方式访问**同一 32 字节内存段中的元素，就会发生合并内存访问。

换句话说，实现合并内存访问的最佳方法是**最大化使用字节数与传输字节数的比率**。

> **确保全局内存访问的适当合并是编写高性能 CUDA 内核时最重要的性能考量之一。应用程序必须尽可能高效地使用内存系统。**

### **使用全局内存的矩阵转置示例**

考虑一个**原地外矩阵转置内核**，该内核将一个大小为 N×N 的 32 位浮点方阵从矩阵 `a` 转置到矩阵 `c` 。

![image.png](2%202%20%E7%BC%96%E5%86%99CUDA%20SIMT%E5%86%85%E6%A0%B8/image%202.png)

此示例采用二维网格，假设启动大小为 32×32 线程的二维线程块，即 `blockDim.x = 32`  和 `blockDim.y = 32` ，因此每个二维线程块将处理矩阵的一个 32×32 图块。每个线程操作矩阵中的唯一元素，因此无需显式同步线程。

```cpp
/* macro to index a 1D memory array with 2D indices in row-major order */
/* ld is the leading dimension, i.e. the number of columns in the matrix */

#define INDX(row, col, ld) ((row * ld) + col)

/* CUDA kernel for naive matrix transpose */
__global__ void naive_cuda_transpose(int m, float*a, float*c)
{
		int myCol = blockDim.x * blockIdx.x + threadIdx.x;
		int myRow = blockDim.y * blockIdx.y + threadIdx.y;
		
		if(myCol < m && myRow < m)
		{
				c[INDX(myCol, myRow, m)] = a[INDX(myRow, myCol, m)];
		}
		return;
}

```

- 在二维线程块中， `x` 索引变化最快，因此连续的 `threadIdx.x` 值应当访问**连续的内存元素**。 `threadIdx.x` 出现在 `myCol` 中，可以观察到当 `myCol` 作为 `INDX` 宏的第二个参数时，连续线程正在读取 `a` 的连续值，因此对 `a` 的读取是**完全合并**的。
- 然而，对 `c` 的写入**并未合并**，因为连续的 `threadIdx.x` 值正在向 `c` 写入彼此间隔 `ld` （主维度）个元素的存储单元。这是由于当前 `myCol` 作为 `INDX` 宏的第一个参数，当 `INDX` 的第一个参数递增 1 时，**内存地址将偏移** `ld` 。当 `ld` 大于 32 时（只要矩阵尺寸超过 32 就会发生），就会达成最坏案例。**【ld > 8, 意味着超出了每次memory transaction的32字节/4字节=8元素的请求范围，导致最大化内存事务数量；ld>32时线程间隔超出每个线程束的线程数量，导致合并访问彻底失效】**

为缓解这些非合并写入，可采用**共享内存技术**

### 共享内存访问模式 Shared Memory Access Patterns

共享内存有 **32 个存储体（banks）**，其组织结构使得连续的 32 位字映射到连续的存储体。“**每个存储体”“每时钟周期”**提供 **32 位带宽**。

当**同一线程束中的多个线程**尝试**访问同一存储体**中的不同元素时，就会发生**存储体冲突（bank conflict）**。

这种情况下，对该存储体中数据的访问将被**序列化**，直到该存储体中的数据被**所有请求线程获取完毕**。这种访问的序列化会导致**性能损失**。

- 两种例外：当同一线程束中的多个线程**访问（读取或写入）同一共享内存位置**时。对于读取访问，**数据字将广播给所有请求线程**；**【指访问同一存储体的同一元素】**
- 对于**写入访问**，**每个共享内存地址**仅由**其中一个线程执行写入操作**（具体由哪个线程执行写入是不确定的）。**【只进行其中一个线程的写入数据，只写一次】**

- **使用共享内存的矩阵转置示例**
    
    本例将共享内存视为**用户管理的缓存**，用于暂存来自全局内存的加载和存储操作，从而实现读写双方的合并全局内存访问。
    
    ```cpp
    /* definitions of thread block size in X and Y directions */
    #define THREADS_PER_BLOCK_X 32
    #define THREADS_PER_BLOCK_Y 32
    
    /* macro to index a 1D memory array with 2D indices in row-major order */
    /* ld is the leading dimension, i.e. the number of columns in the matrix */
    
    #define INDX(row, col, m) ((row * ld) + (col))
    
    /* CUDA kernel for shared memory matrix transpose */
    
    __global__ void smem_cuda_transpose(int m, float* a, float* c)
    {
    		/* declared a statically allocated shared memory array */
    		__shared__ float smemArray[THREADS_PER_BLOCK_X][THREADS_PER_BLOCK_Y];
    		
    		/* determine my row tile and column tile index */
    		const int tileCol = blockDim.x * blockIdx.x;
    		const int tileRow = blockDim.y * blockIdx.y;	
    		
    		/* read from global memory into shared memory array */
    		smemArray[threadIdx.x][threaedIdx.y] = a[INDX(tileRow + threadIdx.y, tileCol + threadIdx.x, m)];
    		
    		/* synchronize the thread in the thread block */
    		_syncthreads();
    		
    		/* write the result from shared memory to global memory */
    		c[INDX(tileCol + threadIdx.y, tileRow + threadIdx.x, m)] = smemArray[threadIdx.y][threadIdx.x];
    		return;
    }
    
    ```
    
- **例证数组检查**
    
    ```cpp
    /* definitions of thread block size in X and Y */
    #define THREADS_PER_BLOCK_X 32
    #define THREADS_PER_BLOCK_Y 32
    
    /* macro to index a 1D memory array with 2D indices in column-major order */
    /* ld is the leading dimension, i.e. the number of rows in the matrix */
    #define INDX(row, col, m) ((col * ld) + (row))
    
    /* CUDA kernel for shared memory matrix transpose */
    
    __global__ void smem_cuda_transpose(int m, float* a, float* c)
    {
    		/* declared a statically allocated shared memory array */
    		__shared__ float smemArray[THREADS_PER_BLOCK_X][THREADS_PER_BLOCK_Y];
    		
    		/* determined my row and column indices for the error checking code */
    		const int myCol = BlockDim.y * BlockIdx.y + threadIdx.y;
    		const int myRow = BlockDim.x * BlockIdx.x + threadIdx.x;
    		
    		/* determined my row tile and column tile index */
    		const int tileY = BlockDim.y * BlockIdx.y;
    		const int tileX = BlockDim.x * BlockIdx.x;
    		
    		if(myRow < m && myCol < m)
    		{
    				/* read from global memory into shared memory array*/ 
    				smemArray[threadIdx.x][threadIdx.y] = a[INDX(tileX + threadIdx.x, tileY + threadIdx.y, m)];
    		}
    		/* synchronize the threads in the thread block */
    		__syncthreads();
    		
    		if(myRow < m && myCol < m)
    		{
    				c[INDX(tileY + threadIdx.x, tileX + threadIdx.y, m)] = smemArray[threadIdx.y][threadIdx.x];
    		}
    		return;
    		
    ```
    

在执行复制操作之前，每个线程会计算其 `tileRow` 和 `tileCol` 索引。这些**索引对应**即将操作的**特定矩阵块**，且这些块索引基于当前执行的线程块确定。

同一线程块中的所有线程具有**相同**的 `tileRow` 和 `tileCol` 值，因此可将其视为该线程块即将操作的矩阵块的**起始位置**。

- 共享内存用于从全局内存中暂存数据，以确保对全局内存的读取和写入都能正确合并。
- 共享内存使得同一线程块内的线程能够相互共享数据。

### 共享内存体冲突 Shared Memory Bank Conflicts

在前述矩阵转置示例中，虽然实现了对全局内存的正确合并访问，但并**未考虑是否存在共享内存体冲突**。考虑以下二维共享内存声明：

`__shared__ float smemArray[32][32];`

由于一个 warp 包含 32 个线程，同一个 warp 中的每个线程将具有固定的 `threadIdx.y` 值，并拥有 `0 <= threadIdx.x < 32` 。

![image.png](2%202%20%E7%BC%96%E5%86%99CUDA%20SIMT%E5%86%85%E6%A0%B8/image%205.png)

左图展示了 warp 中的线程访问 `smemArray` 数据列时的情况。Warp 0 正在访问从 `smemArray[0][0]` 到 `smemArray[31][0]` 的内存地址。按照 C++多维数组的存储顺序，最后一个索引变化最快，因此 warp 0 中连续的线程访问的内存地址间隔 32 个元素。如图所示，**不同颜色代表不同的存储体**，warp 0 对整个数据列的访问将导致 **32 路存储体冲突**。

右图展示了 warp 中的线程访问 `smemArray` 数据行时的情况。Warp 0 正在访问从 `smemArray[0][0]` 到 `smemArray[0][31]` 的内存地址。这种情况下，warp 0 中连续的线程访问的是相邻的内存地址。**理想情况下**，warp 中的**每个线程应当访问具有不同颜色标识**的共享内存地址。

---

避免存储体冲突的常见解决方案是通过为数组**列维度增加填充位（padding）来扩展共享内存**，具体实现如下：

```cpp
__shared__ float smemArray[THREADS_PER_BLOCK_X][THREADS_PER_BLOCK_Y+1];
```

![image.png](2%202%20%E7%BC%96%E5%86%99CUDA%20SIMT%E5%86%85%E6%A0%B8/image%206.png)

对 `smemArray` 声明的这一微小调整将消除存储体冲突。如图，共享内存数组声明为 32×33 尺寸时可见，无论同一线程束中的线程沿整列还是整行访问共享内存数组，存储体冲突均已消除。

<aside>
💡

**为什么列数+1能防止bank conflict？**

shared memory有32个bank，所以分**配的bank号 = 共享内存地址 % 32**

同一线程每次访问共享内存的地址间隔恰好都是32，根据上述公式访问的bank号是相同的，且共享内存地址又不同，所以会导致32路的存储体冲突（bank conflict）。如果列数+1,则刚好同一线程访问的下一个共享内存的bank号偏移1，避开了冲突。

**注意：全局内存才需要注意连续线程访问，而共享内存需要注意存储体冲突。**

</aside>

## 原子操作 Atomics

高效的 CUDA 核心依赖于尽可能多地表达算法的**并行性**。GPU 核心执行的**异步特性**要求线程尽可能**独立运行**。**【理解：核函数是并行执行的，但各线程需尽量独立而不是互相依赖（阻塞排队）才高效】**

在整个网格层面，不存在同步网格内所有线程的机制。通过**原子函数（atomic function）**可以实现对**全局内存**位置的**同步访问**。

原子函数允许线程**锁定**全局内存**位置**，并对该位置执行**读取-修改-写入**操作。

当**锁被持有**时，其他线程**无法访问同一内存位置**。**【理解：防止数据竞争】**

- CUDA 提供的原子函数具有与 C++标准库原子操作**相同的行为**，如  `cuda::std::atomic`  和 `cuda::std::atomic_ref` 。
- CUDA 还提供**扩展的 C++原子操作** `cuda::atomic` 和 `cuda::atomic_ref` ，允许用户指定原子操作的线程作用域。

下面是一个使用 `cuda::atomic_ref` 执行设备级原子加法操作的示例：

```cpp
__global__ void sumReduction(int n, float* array, float* result)
{
		tid = threadIdx.x + BlockDim.x * BlockIdx.x;
		// atomic声明的是原子类型，但'result'是指针，无法改变定义
		// atomic_ref作用是创建指针'result'的原子引用'result_ref' 
		cuda::atomic_ref<float, cuda::thread_scope_device> result_ref(result);
		result_ref.fetch_add(array[tid]);
}
```

<aside>
💡

其中 `array` 是一个浮点数数组， `result` 是一个指向全局内存中某位置的浮点指针，该位置将用于存储数组求和的结果。

原子函数应谨慎使用，因为它们会**强制线程同步**，从而可能影响性能。

</aside>

## 协作组 Cooperative Group

协作组是 CUDA C++中提供的一种**软件工具**，允许应用程序定义可**相互同步**的**线程组**，即使该线程组跨越多个线程块、单个 GPU 上的多个网格，甚至跨越多块 GPU。

## 内核启动与占用率 Kernel Launch and Occupancy

当 CUDA 内核启动时，CUDA 线程会根据**内核启动时指定的执行配置**，分组为**线程块**和**网格**。

内核启动后，调度器将**线程块分配**给**流多处理器**。

具体哪些线程块在哪些流多处理器上执行，应用程序无法控制或查询，**调度器**也不提供任何顺序保证。因此，程序**不能依赖特定的调度顺序或方案**来确保正确执行。**【补充：内核首次启动时，只要SM还有可用硬件资源，调度器就会不断分配线程块，若某一时刻所有SM均无容量，则会等待SM完成先前分配的线程块，直至所有线程块均完成调度与执行。】**

---

`cudaGetDeviceProperties` 函数允许应用程序通过设备属性查询每个 SM 的资源上限。需注意相关限制同时存在于 SM 级别与线程块级别。

- `maxBlocksPerMultiProcessor` ：每个 SM 可驻留的线程块数量上限。
- `sharedMemPerMultiprocessor` ：每个 SM 可用的共享内存容量（以字节为单位）。
- `regsPerMultiprocessor` ：每个流多处理器（SM）可用的 32 位寄存器数量。
- `maxThreadsPerMultiProcessor` ：每个流多处理器（SM）可驻留的最大线程数量。
- `sharedMemPerBlock` ：单个线程块可分配的共享内存最大容量（以字节为单位）。
- `regsPerBlock` ：单个线程块可分配的最大 32 位寄存器数量。
- `maxThreadsPerBlock` ：每个线程块的最大线程数。

CUDA 内核的占用率是指活跃线程束数量与流多处理器支持的最大活跃线程束数量之比【对象是线程束】。通常而言，保持尽可能高的占用率是一种良好实践，这有助于隐藏延迟并提升性能。

| Resource   | Value   |
| --- | --- |
| `maxBlocksPerMultiProcessor` | 32 |
| `sharedMemPerMultiprocessor` | 233472 |
| `regsPerMultiprocessor` | 65536 |
| `maxThreadsPerMultiProcessor` | 2048 |
| `sharedMemPerBlock` | 49152 |
| `regsPerBlock` | 65536 |
| `maxThreadsPerBlock` | 1024 |

上表为计算能力为10.0的设备的SM 资源示例

<aside>
💡

如果一个内核以 `testKernel<<<512, 768>>>()` 参数启动，即每个线程块包含 768 个线程，由于 `maxThreadsPerMultiProcessor` 参数设定为 2048，调度器只能为SM分配2个线程块。因此占用率计算公式为：(768 * 2) / 2048，即 75%。

如果内核以 `testKernel<<<512, 32>>>()` 形式启动，即每个线程块包含 32 个线程，则每个 SM 不会遇到 `maxThreadsPerMultiProcessor` 的限制。但由于 `maxBlocksPerMultiProcessor` 为 32，调度器仅能为每个 SM 分配 32 个线程块，驻留在 SM 上的总线程数将是 32 个线程块 × 每块 32 线程，共计 1024 个线程，此情形下的占用率为 1024/2048，即 50%。

同样的分析可应用于共享内存场景。例如，若某个内核使用 100KB 共享内存，则调度器最多只能为每个 SM 分配 2 个线程块——因为该 SM 上的第三个线程块需要额外 100KB 共享内存，总量将达 300KB，这已超过每个 SM 可用的 233472 字节容量。

</aside>
