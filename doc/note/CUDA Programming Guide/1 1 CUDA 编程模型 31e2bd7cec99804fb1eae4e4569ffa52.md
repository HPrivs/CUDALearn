# 1.1 CUDA 编程模型

CUDA 编程模型基于异构计算系统构建，这意味着该系统同时包含 GPU 和 CPU。

直接与 CPU 连接的内存被称为主机内存，而 CPU 本身则被称为主机。同理，直接与 GPU 连接的内存被称为设备内存，GPU 本身则被称为设备。

## GPU硬件模型结构

GPU 可视为由**流式多处理器**（Streaming Multiprocessor, SM）构成的集合，这些处理器被组织成称为图形处理集群（GPC）的组。每个 SM 包含**寄存器文件**、**统一数据缓存**以及若干执行计算的功能单元。统一数据缓存为**共享内存和 L1 缓存**提供物理资源，其在 L1 缓存与共享内存间的分配比例可在运行时配置。

![image.png](1%201%20CUDA%20%E7%BC%96%E7%A8%8B%E6%A8%A1%E5%9E%8B/image.png)

GPU 包含许多流式多处理器（SM），每个 SM 由众多功能单元组成。图形处理集群（GPC）是多个 SM 的集合。GPU 则是通过 GPU 内存相互连接的一组 GPC 集合。CPU 通常包含若干核心及连接系统内存的内存控制器。CPU 与 GPU 通过 PCIe 或 NVLINK 等互连技术相连。

## 线程块和网格 Thread Blocks and Grids

当应用程序启动内核时，会同时启动大量线程（通常达数百万个）。这些线程被组织成块，称为线程块。线程块进一步组织成网格结构，**同一网格中的所有线程块具有相同的尺寸和维度**。

![image.png](1%201%20CUDA%20%E7%BC%96%E7%A8%8B%E6%A8%A1%E5%9E%8B/image%201.png)

<aside>
💡

线程块网格示意图。每个箭头代表一个线程

</aside>

线程块和网格可以是一维、二维或三维的。这些维度有助于简化将单个线程映射到工作单元或数据项的过程。线程块的每个线程都具备唯一标识，能确定执行内核的线程自身所属线程块位置和所属线程块在所在网格的位置。**【线程通过内置变量获得位置和维度信息】**

一个线程块中的所有线程都在同一个流多处理器（SM）中执行。这使得线程块内的线程能够高效地进行通信与同步。线程块内的所有线程均可访问片上共享存储器，该存储器可用于线程块内各线程间的信息交换。

![image.png](1%201%20CUDA%20%E7%BC%96%E7%A8%8B%E6%A8%A1%E5%9E%8B/image%202.png)

<aside>
💡

 每个流式多处理器（SM）包含一个或多个活跃线程块。在本示例中，每个 SM 同时调度了三个线程块。网格中的线程块被分配到各个 SM 的顺序不保证任何特定顺序。

</aside>

CUDA 编程模型支持任意规模的网格在任何规模的 GPU 上运行，无论该 GPU 仅有一个流式多处理器还是拥有数千个流式多处理器。为实现这一特性，除少数例外情况，CUDA 编程模型要求不同线程块中的线程之间不得存在数据依赖关系。换言之，线程不应依赖同一网格中不同线程块内其他线程的计算结果，也不应与这些线程进行同步操作。同一线程块内的所有线程始终在同一流式多处理器上并发执行，而网格中的不同线程块则在可用流式多处理器间动态调度，其执行顺序可以是任意的。**【1、不同线程块之间不得存在数据依赖关系；2、不同线程块在SM中动态调度，执行顺序是任意的。】**

### 线程块簇 Thread Block Clusters

除了线程块之外，具备计算能力 9.0 及更高版本的 GPU 还支持一个可选的层级分组结构——集群。

集群是由多个线程块组成的集合，与线程块和网格类似，其布局可采用一维、二维或三维形式。

![image.png](1%201%20CUDA%20%E7%BC%96%E7%A8%8B%E6%A8%A1%E5%9E%8B/image%203.png)

<aside>
💡

当指定集群时，线程块不仅在网格中处于相同位置，且在所属集群中亦有相应位置。

</aside>

由于这些线程块在单个 GPC 内被同时调度，位于不同线程块但属于同一集群的线程可通过协作组（Cooperative Groups）提供的软件接口相互通信和同步。**【同一集群的线程可通过软件接口通信和同步】**

![image.png](1%201%20CUDA%20%E7%BC%96%E7%A8%8B%E6%A8%A1%E5%9E%8B/image%204.png)

<aside>
💡

当指定集群时，集群中的线程块会按照其集群形状排列在网格内。集群内的所有线程块会在单个图形处理集群（GPC）的流式多处理器（SM）上同时调度。**【指定的1x2的线程块集群被单个GPC同时调度】**

</aside>

### 线程束与SIMT Warps and SIMT

在线程块内部，线程被组织成若干组，**每组包含 32 个线程**，称为线程束（warp）。

线程束以**单指令多线程**（Single Instruction Multiple-Threads, SIMT）范式执行内核代码。

当线程由线程束执行时，它们被**分配一个线程束通道**。线程束通道的**编号从 0 到 31**，并且线程块中的线程会按照硬件多线程中详细说明的可预测方式分配到线程束中。

线程束中的所有线程同时执行相同的指令。如果线程束中的某些线程在执行过程中遵循某个控制流分支，而其他线程不遵循，那么**不遵循该分支的线程将被屏蔽**，而遵循该分支的线程则继续执行。

![image.png](1%201%20CUDA%20%E7%BC%96%E7%A8%8B%E6%A8%A1%E5%9E%8B/image%205.png)

<aside>
💡

当线程束中的不同线程遵循不同的代码路径时，有时称为**线程束分化（Warp Divergence）**。由此可见，**当线程束内的线程遵循相同的控制流路径时，GPU 的利用率达到最大**。

</aside>

在 SIMT 模型下，所有线程束中的线程都以**同步锁步**的方式在核函数中推进。**【同步锁步指的是线程束中的所有线程在同一时刻必须执行同一条指令。】**

CUDA 编程模型与 SIMT 机制规定线程束中的所有线程必须同步执行代码。

线程束执行的一个含义是，最佳做法是将线程块的总线程数指定为 32 的倍数。使用任意数量的线程都是合法的，但当总线程数不是 32 的倍数时，线程块的最后一个线程束在整个执行过程中将有部分通道处于未使用状态。这可能导致该线程束的功能单元利用率和内存访问效率达不到最优水平。**【线程块的总线程数最好为32的倍数，不然执行过程中，最后一个线程束可能会有部分通道处于未使用状态，利用率效率受限。】**

## GPU内存

**异构系统**拥有**多层级内存空间**，而图形处理器除了**缓存**之外，还配备了多种类型的**可编程片上内存**。

### **异构系统中的 DRAM 内存**

从设备代码的角度来看，**GPU 连接的 DRAM 被称为全局内存**，因为 GPU 内的所有流式多处理器均可访问它。这一术语并不意味着该内存在系统各处都能被访问。与 **CPU 连接的 DRAM 则被称为系统内存或主机内存。**

与 CPU 类似，GPU 同样采用**虚拟内存寻址机制**。在目前所有受支持的系统中，**CPU 与 GPU 共享统一的内存虚拟寻址空间**。这意味着系统中每个 GPU 的**虚拟内存地址范围都是独立且唯一的**，既与 CPU 的地址空间区隔，也区别于系统中其他 GPU 的地址范围。

### **GPU 片上内存**

![image.png](1%201%20CUDA%20%E7%BC%96%E7%A8%8B%E6%A8%A1%E5%9E%8B/image%206.png)

除了全局内存外，每颗 GPU 还具备**部分片上内存**。每个流式多处理器（SM）拥有**独立的寄存器文件与共享内存**。这些内存属于 SM 的组成部分，可由该 SM 内部执行的线程极速访问，但其他 SM 中运行的线程无法访问。

**寄存器文件存储**通常由编译器分配的**线程局部变量**。**共享内存**可供**线程块或线程簇内的所有线程访问**，可用于线程块或线程簇内部的数据交换。**【理解：寄存器文件只能由受分配的线程访问（线程私有），而共享内存能在线程块或线程簇内的所有线程访问】**

SM 中的寄存器文件与统一数据缓存**具有固定容量**。

寄存器文件、共享内存空间和 L1 缓存由线程块内的**所有线程共享**。

要将一个线程块调度至 SM 执行，**线程块内每个线程所需寄存器总数**乘以**线程块中的线程数量**必须**小于或等于** SM 中**可用的寄存器数量**。

共享内存分配在线程块级别进行。

### **缓存 Caches**

每个流多处理器（SM）拥有**独立的 L1 缓存**，该缓存属于统一数据缓存的一部分。而更大的 L2 缓存则由 GPU 内**所有流多处理器共享**。**【见图】**

![image.png](1%201%20CUDA%20%E7%BC%96%E7%A8%8B%E6%A8%A1%E5%9E%8B/image%207.png)

每个流多处理器还设有独立的常量缓存，用于缓存全局内存中声明在**内核生命周期内保持恒定的数值**。

## **统一内存 Unified Memory**

当应用程序在 GPU 或 CPU 上显式分配内存时，该**内存仅对运行在该设备上的代码可访问**。也就是说，CPU 内存只能被 CPU 代码访问，而 GPU 内存只能由运行在 GPU 上的内核访问**【理解：就像PyTorch编程时CPU的tensor无法与GPU的tensor一同计算】**

**统一内存**的 CUDA 特性允许应用程序进行内存分配，使得这些内存既可从 CPU 也可从 GPU 访问。即使采用统一内存，仍需通过**最小化内存迁移**、并**尽可能从直连内存所在位置的处理器直接访问**数据，才能实现最佳性能。**【理解：只是自动化了内存迁移，不代表消除了物理间的数据传输。】**
