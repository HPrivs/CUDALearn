# 2.1 CUDA C++

## 核函数 Kernels

主机端调用并在 GPU 上执行的函数称为核函数。核函数被设计为由众多并行线程同时执行。

### 核函数的定义

内核的代码使用 `__global__` 声明说明符来指定。这向编译器表明，**此函数将被编译为可在 GPU 上执行的方式**，并允许通过内核启动来调用。**内核启动**是一种启动内核运行的操作，通常从 **CPU 发起**。内核是**返回类型**为 `void` 的函数。

```jsx
// Kernel definition
__global__ void vecAdd(float* A, float* B, float* C)
{

}
```

### 内核函数启动

并行执行内核的**线程数量指定**为内核启动的一部分。这被称为**执行配置**。同一内核的不同调用可能使用不同的执行配置，例如**不同数量的线程或线程块**。

有两种从 CPU 代码启动内核的方法：**三重尖括号表示法**和 `cudaLaunchKernelEx` 。

### **三重尖括号表示法 Triple Chevron Notation**

三重尖括号表示法是CUDA C++ 语言扩展，**最常用**的内核启动方式。它使用三个尖括号字符来封装内核启动的执行配置，即 `<<< >>>` 。执行配置参数在尖括号内**以逗号分隔的列表形式指定。**

```jsx
__global__ void vecAdd(float* A, float* B, float* C)
 {

 }

int main()
{
    ...
    // Kernel invocation
    vecAdd<<<1, 256>>>(A, B, C);
    ...
}
```

三重尖括号符号的前两个参数分别代表**网格维度**和**线程块维度**。在使用一维线程块或网格时，可直接用整数指定维度。

<aside>
💡

1. **网格维度（Grid Dimension）**：指定有多少个线程块
2. **线程块维度（Block Dimension）**：指定每个线程块中有多少个线程

上述代码启动了一个包含 256 个线程的线程块。

</aside>

**每个块中的线程数量存在上限**，这是因为一个块的所有线程都位于**同一个流式多处理器（SM）上**，并且**必须共享该 SM 的资源**。在**当前 GPU 架构中**，一个线程块**最多可包含 1024 个线程**。若资源允许，单个 SM 上可同时调度多个线程块。

内核启动相对于主机线程是**异步**的。也就是说，内核将被配置在 GPU 上执行，但主机代码在继续执行前**不会等待内核完成（甚至不会等待其开始）**在 GPU 上的执行。

```jsx
int main(){
...
dim3 grid(16,16);
dim3 block(8,8);
MatAdd<<<grid, block>>>(A,B,C);
...
}
```

<aside>
💡

当使用**二维或三维网格或线程块**时，**CUDA 类型** `dim3` 被用作**网格和线程块的维度参数**。代码片段展示了一个 `MatAdd` 内核的启动示例：使用 16×16 的线程块网格，每个线程块为 8×8 规格。

</aside>

### 线程与网格索引内置函数 Thread and Grid Index Intrinsics

在内核代码中，CUDA 提供**内置函数**以**访问**执行配置参数及线程或块的**索引**。

- `threadIdx` 用于获取**线程在其所属线程块内的索引**，同一线程块中的每个线程均具有不同的索引值。
- `blockDim`  提供了**线程块的维度**，该维度在内核启动的执行配置中指定。
- `blockIdx` 用于获取**线程块在网格中的索引**，每个线程块在网格中具有唯一的索引标识。
- `gridDim` 提供了**网格的维度**，该维度在内核启动时的执行配置中指定。

这些内建向量都是三维向量，包含 `.x` 、 `.y` 和 `.z` 三个成员分量。启动配置**未指定的维度将默认设为 1**。 

`threadIdx` 和 `blockIdx` 采用**零基索引**，即 `threadIdx.x` 的取值范围是从 0 到 `blockDim.x-1` （包含边界值）。 `.y` 和 `.z` 在各自维度上的运作方式与此相同。

同样， `blockIdx.x` 的取值将从 0 到 `gridDim.x-1` （含 `gridDim.x-1` ），而 `.y` 和 `.z` 维度也分别遵循相同的规律。

这些参数允许**单个线程识别其应执行的工作任务**。

```jsx
__global__ void VecAdd(float* A, float* B, float* C)
{
		// calculate which element this thread is responsible for computing
		int workIndex = threadIdx.x + blockDim.x * blockIdx.x
		// Perform computation
		C[workIndex] = A[workIndex] + B[workIndex];
}

int main()
{
		...
		VecAdd<<<4, 256>>>(A, B, C);
		...
}
```

该内核接收三个参数，每个参数都是浮点数向量。该内核对 `A` 和 `B` 进行逐元素加法运算，并将结果存储在 `C` 中。内核通过**并行化设计使每个线程执行一次加法运算**，具体计算哪个元素则由其线程索引与网格索引共同决定。

<aside>
💡

使用 4 个包含 256 个线程的线程块对包含 1024 个元素的向量执行加法运算。

- 在第一个线程块中， `blockIdx.x` 的值为零，因此每个线程的 workIndex 将直接等于其 `threadIdx.x` 。
- 在第二个线程块中， `blockIdx.x` 的值为 1，此时 `blockDim.x * blockIdx.x` 与 `blockDim.x` 相，第二个线程块中每个线程的 `workIndex` 将对应其 `threadIdx.x + 256` 。
- 在第三个线程块中， `workIndex` 的值将为 `threadIdx.x + 512` 。

**扩展到二维或三维时**，往往**在每个维度上都遵循相同的模式**。

</aside>

### 边界检查 Bound Checking

为使内核能够**处理任意长度的向量**，我们可**添加检查**以确保内存访问**未超出数组边界**，如下所示，随后启动一个包含部分**非活跃线程**的线程块。

```cpp
__global__ void VecAdd(float* A, float* B, float* C, int vectorLength)
{
		// calculate which element this thread is responsible for computing
		int workIndex = threadIdx.x + blockDim.x * blockIdx.x
			
		if (workIndex < vectorLength)
		{
				// Perform computation
				C[workIndex] = A[workIndex] + B[workIndex];
		}
}
```

当 `workIndex` 超过 `vectorLength` 时，线程将自动退出且不执行任何操作。

虽然启动块内闲置线程不会产生显著开销，但**仍应避免启动完全无工作线程的线程块**。此内核现可处理向量**长度不为块大小整数倍**的情况。

**所需线程块的数量**可通过将**所需线程数**（本例中为向量长度）除以**每个块的线程数**，然后**取上整数**来计算。

```cpp
// vectorLength is an integer storing number of elements in the vector
int threads = 256;
int blocks = (vectorLength + threads - 1) / threads;
vecAdd<<<blocks, threads>>>(A, B, C, vectorLength);
```

**CUDA 核心计算库（CUDA Core Compute Library, CCCL）**提供了一个便捷工具 `cuda::ceil_div` ，用于执行向上取整除法以计算内核启动所需的块数量。通过包含头文件 `<cuda/cmath>` 即可使用该工具。

```cpp
int threads = 256;
int blocks = cuda::ceil_div(vectorLength, threads);
vecAdd<<<blocks, threads>>>(A, B, C, vectorLength);
```

## GPU计算中的内存管理 Memory in GPU Computing

为了使用上述的 `vecAdd` 内核，数组 `A` 、 `B` 和 `C` 必须位于 GPU 可访问的内存中。实现这一目标有几种不同方法，本文将重点介绍**其中两种**。

### 统一内存 Unified Memory

统一内存是 CUDA 运行时的一项功能，它使 NVIDIA 驱动程序能够管理**主机（host）**与**设备（device）**之间的数据移动。

内存可通过 `cudaMallocManaged` API 分配，或通过使用 `__managed__` 说明符声明变量来分配。NVIDIA 驱动程序将确保每当 GPU 或 CPU 尝试访问内存时，内存均可被访问。

```cpp
void UnifiedMemExample(int vectorLength)
{
		// Pointers to memory vectors
		float* A = nullptr;
		float* B = nullptr;
		float* C = nullptr;
		float* comparisionResult = (float*)malloc(vectorLength * sizeof(float));

		// Use unified memory to allocate buffers
		cudaMallocManaged(&A, vectorLength * sizeof(float));
		cudaMallocManaged(&B, vectorLength * sizeof(float));
		cudaMallocManaged(&C, vectorLength * sizeof(float));
		
		// Initialize vectors on the host
		InitArray(A, vectorLength);
		InitArray(B, vectorLength);
		
		// Launch the kernel. Unified memory will make sure A, B and C are 
		// accesible to the GPU
		int threads; 
		int blocks = cuda::ceil_div(vectorLength, threads);
		VecAdd<<<blocks, threads>>>(A, B, C, vectorLength);
		// Wait for the kernel to complete execution
		cudaDeviceSynchronize();
		
		// Perform computation serially on CPU for comparision
		serialVecAdd(A, B, comparisionResult, vectorLength);
		
		// Confirm that CPU and GPU got the same answer
		if (vectorApproximatelyEqual(C, comparisionResult, vectorLength))
		{
				printf("Unified Memory: CPU and GPU answers match\n");
		}
		else
		{
				printf("Unified Memory: Error - CPU and GPU answers do not match\n");
		}
		
		// Clean up
		cudaFree(A);
		cudaFree(B);
		cudaFree(C);
		free(comparisionResult);
}
```

<aside>
💡

代码展示了一个完整的函数，用于启动将使用 GPU 的输入和输出向量的 `vecAdd` 内核。 `cudaMallocManaged` 分配可从 CPU 或 GPU 访问的缓冲区。这些缓冲区通过 `cudaFree` 释放。

</aside>

在某些 Linux 系统上（例如**具备地址转换服务**或**异构内存管理功能**的系统），所有系统内存会**自动成为统一内存**，此时无需使用 `cudaMallocManaged` 标识符或 `__managed__` 说明符。

### 显式内存管理 Explicit Memory Management

显式管理内存分配和内存空间之间的数据迁移**有助于提高应用程序性能**，尽管这会使代码更加冗长。下面的代码使用 `cudaMalloc` 显式地在 GPU 上分配内存。

```cpp
void explicitMemExample(int vectorLength)
{
		// Pointers for host memory
		float* A = nullptr;
		float* B = nullptr;
		float* C = nullptr;
		float* comparisionResult = (float*)malloc(vectorLength * sizeof(float));
		
		// Pointers for device memory
		float* devA = nullptr;
		float* devB = nullptr;
		float* devC = nullptr;
		
		// Allocate Host Memory using cudaMallocHost API. This is the best practice
		// when buffers will be used for copies between CPU and GPU memory.
		cudaMallocHost(&A, vectorLength * sizeof(float));
		cudaMallocHost(&B, vectorLength * sizeof(float));
		cudaMallocHost(&C, vectorLength * sizeof(float));
		
		// Initialization vectors on the host
		InitArray(A, vectorLength);
		InitArray(B, vectorLength);
		
		// Start-allocate-and-copy
		// Allocate memory on the GPU
		cudaMalloc(&devA, vectorLength * sizeof(float), cudaMemcpyDefault);
		cudaMalloc(&devB, vectorLength * sizeof(float), cudaMemcpyDefault);
		cudaMalloc(&devC, vectorLength * sizeof(float), cudaMemcpyDefault);
		
		// Copy data to the GPU
		cudaMemcpy(devA, A, vectorLength * sizeof(float), cudaMemcpyDefault);
		cudaMemcpy(devB, B, vectorLength * sizeof(float), cudaMemcpyDefault);
		cudaMemset(devC, 0, vectorLength * sizeof(float));
		// end allocate-and-copy
		
		// Launch the kernel
		int threads = 256;
		int blocks = cuda::ceil_div(vectorLength, threads);
		vecAdd<<<blocks, threads>>>(devA, devB, devC);
		// Wait for kernel excution to complete
		cudaDeviceSynchronize();
		
		// Copy result to the host
		cudaMemcpy(C, devC, vectorLength * sizeof(float), cudaMemcpyDefault);
		
		// Perform computation serailly on CPU for comparision
		serialVecAdd(A, B, comparisionResult, vectorLength);
		
		// Confirm that CPU and GPU got the same answer
		if (vectorApproximatelyEqual(C, comparisionResult, vectorLength))
		{
				printf("Explicit Memory: CPU and GPU answers match\n");
		}		
		else
		{
				printf("Explicit Memory: Error - CPU and GPU answers to not match\n");
		}
		
		cudaFree(devA);
		cudaFree(devB);
		cudaFree(devC);
		cudaFreeHost(A);
		cudaFreeHost(B);
		cudaFreeHost(C);
		free(comparisionResult);
}
```

CUDA API `cudaMemcpy` 用于将数据从驻留在 **CPU 上的缓冲区**复制到驻留在 **GPU 上的缓冲区**。

<aside>
💡

除了目标指针、源指针和字节大小之外， `cudaMemcpy` 的最后一个参数是 `cudaMemcpyKind_t` 。

该参数可赋予不同取值，例如：从 CPU 复制到 GPU 时使用 `cudaMemcpyHostToDevice`或在 GPU 内部或 GPU 之间复制时使用 `cudaMemcpyDeviceToDevice` 。

`cudaMemcpyDefault` 作为最后一个参数传递给 `cudaMemcpy` 。这将导致 CUDA 根据源指针和目标指针的值来确定要执行的复制类型。

</aside>

`cudaMemcpy` API 是**同步的**，即在**复制完成前不会返回控制权**。

`cudaMallocHost` 在 CPU 上分配内存。这将**在主机端分配页锁定内存（page-locked memory）**，从而提升复制性能，并且是**异步内存传输的必要条件**。**【注：页锁定内存指无论如何都不会被移动的物理内存。】**

通常最佳实践是：对于将在 GPU 数据收发过程中使用的 CPU 缓冲区，建议使用页锁定内存。若主机端页锁定内存过多，部分系统的性能可能下降。最佳实践是仅对需要向 GPU 发送或接收数据的缓冲区进行页锁定。**【理解：一般仅对需要频繁进行GPU数据收发的内存才分配页锁定内存。页锁定内存分配速度慢，不用于频繁分配释放。】**

## CPU与GPU同步 Synchronizing CPU and GPU

**内核启动相对于调用它的 CPU 线程是异步的**。这意味着 CPU 线程的控制流会在内核完成执行之前继续运行，甚至可能在内核实际启动之前就继续执行。

为确保在**主机代码继续执行前内核已完成计算**，**必须采用某种同步机制**。

同步 GPU 与主机线程的最简单方法是使用 `cudaDeviceSynchronize` ，该方法会**阻塞主机线程**，直到 GPU 上所有先前提交的工作完成。

若存在多个流在GPU上执行工作，它将**等待所有流中的工作完成**。

此时建议使用**流同步 API** 仅与特定流进行同步，或采用 **CUDA 事件机制**。

---

块内的线程可以通过**共享内存**共享数据，并通过**同步机制来协调内存访问**。

块级同步**最基本的内置机制**是 `__syncthreads()` 指令，该指令作为一个屏障，要求块内**所有线程必须等待直到所有线程准备就绪**后，方可继续执行。

共享内存被设计为**靠近各处理器核心的低延迟内存**（类似于 L1 缓存），且其同步机制需保持**轻量级**。

这种同步**仅作用于单个线程块内的线程**。CUDA 编程模型**不支持线程块之间的同步**，而**协作组（Cooperative Groups）**提供了可设定**超越单个线程块范围的同步域机制**。

通常，将同步控制在单个线程块内可获得最佳性能。线程块仍可通过**原子内存函数**处理共同结果

## 整合应用

简单向量加法内核的**完整代码**，包含所有主机代码及用于核验结果正确性的实用函数。

- 统一内存管理 Unified Memory
    
    ```cpp
    #include <cuda_runtime_api.h>
    #include <memory.h>
    #include <cstdlib>
    #include <ctime>
    #include <stdio.h>
    #include <cuda/cmath>
    
    __global__ void vecAdd(float* A, float* B, float* C, int vectorLength)
    {
    	int workIndex = threadIdx.x + blockDim.x * blockIdx.x;
    	if (workIndex < vectorLength)
    	{
    		C[workIndex] = A[workIndex] + B[workIndex];
    	}
    }
    
    void initArray(float* A, int length)
    {
    	std::srand(std::time({}));
    	for (int i = 0; i < length; i++)
    	{
    		A[i] = rand() / (float)RAND_MAX;
    	}
    }
    
    void serialVecAdd(float* A, float* B, float* C, int vectorLength)
    {
    	for (int i = 0; i < vectorLength; i++)
    	{
    		C[i] = A[i] + B[i];
    	}
    }
    
    bool vectorApproximatelyEqual(float* A, float* B, int length, float epsilon = 0.00001)
    {
    	for (int i = 0; i < length; i++)
    	{
    		if (abs(A[i] - B[i]) > epsilon)
    		{
    			printf("Index %d mismatch: %f != %f", i, A[i], B[i]);
    			return false;
    		}
    	}
    	return true;
    }
    
    // Unified-memory-begin
    void unifiedMemExample(int vectorLength)
    {
    	float* A = nullptr;
    	float* B = nullptr;
    	float* C = nullptr;
    	float* comparisonResult = (float*)malloc(vectorLength * sizeof(float));
    
    	// Use unified memory to allocate buffuers
    	cudaMallocManaged(&A, vectorLength * sizeof(float));
    	cudaMallocManaged(&B, vectorLength * sizeof(float));
    	cudaMallocManaged(&C, vectorLength * sizeof(float));
    
    	// Initialize vectors on the GPU
    	initArray(A, vectorLength * sizeof(float));
    	initArray(B, vectorLength * sizeof(float));
    
    	// Launch the kernel. Unified memory will make sure A, B and C are
    	// accesible to the GPU.
    	int threads = 256;
    	int blocks = cuda::ceil_div(vectorLength, threads);
    	vecAdd <<<threads, blocks>>> (A, B, C, vectorLength);
    	// Wait for the kernel to complete computation.
    	cudaDeviceSynchronize();
    
    	// Perform computation serially on CPU for comparison.
    	serialVecAdd(A, B, comparisonResult, vectorLength);
    
    	// Confirm that CPU and GPU got the same answer.
    	if (vectorApproximatelyEqual(C, comparisonResult, vectorLength))
    	{
    		printf("Unified Memory: CPU and GPU answers match\n");
    	}
    	else
    	{
    		printf("Unified Memory: Error - CPU and GPU answers do not match\n");
    	}
    
    	// Clean Up
    	cudaFree(A);
    	cudaFree(B);
    	cudaFree(C);
    	free(comparisonResult);
    }
    // Unified-memory-end
    
    int main(int argc, char** argv)
    {
    	int vectorLength = 256;
    	if (argc > 2)
    	{
    		vectorLength = std::atoi(argv[1]);
    	}
    	unifiedMemExample(vectorLength);
    	return 0;
    }
    ```

    ```cpp
    	cudaMemset(devC, 0, vectorLength * sizeof(float));
    	
    	// Launch the kernel. Unified memory will make sure A, B and C are
    	// accesible to the GPU.
    	int threads = 256;
    	int blocks = cuda::ceil_div(vectorLength, threads);
    	vecAdd <<<threads, blocks>>> (devA, devB, devC, vectorLength);
    	// Wait for the kernel to complete computation.
    	cudaDeviceSynchronize();
    	cudaMemcpy(C, devC, vectorLength * sizeof(float), cudaMemcpyDefault);
    
    	// Perform computation serially on CPU for comparison.
    	serialVecAdd(A, B, comparisonResult, vectorLength);
    
    	// Confirm that CPU and GPU got the same answer.
    	if (vectorApproximatelyEqual(C, comparisonResult, vectorLength))
    	{
    		printf("Unified Memory: CPU and GPU answers match\n");
    	}
    	else
    	{
    		printf("Unified Memory: Error - CPU and GPU answers do not match\n");
    	}
    
    	// Clean Up
    	cudaFreeHost(A);
    	cudaFreeHost(B);
    	cudaFreeHost(C);
    	cudaFree(devA);
    	cudaFree(devB);
    	cudaFree(devC);
    	free(comparisonResult);
    }
    // Unified-memory-end
    
    int main(int argc, char** argv)
    {
    	int vectorLength = 256;
    	if (argc > 2)
    	{
    		vectorLength = std::atoi(argv[1]);
    	}
    	unifiedMemExample(vectorLength);
    	return 0;
    }
    ```
    

## 运行时初始化 Runtime Initialization

- CUDA 运行时会为系统中的**每个设备**创建一个 **CUDA 上下文**。
- 该上下文是该设备的主上下文，并在**首次调用**激活该设备上下文的运行时函数时初始化。
- 该上下文由应用程序的**所有主机线程共享**。
- 作为上下文创建的一部分，**设备代码**会在必要时进行即时编译并**加载到设备内存**中。
- CUDA 运行时创建的主上下文可通过**驱动 API** 进行访问以实现互操作性

<aside>
💡

`cudaInitDevice` 与 `cudaSetDevice` 函数调用会初始化运行时环境及关联指定设备的主上下文。

若运行时 API 请求早于这些调用发生，系统将**隐式使用设备 0** 并按需**自我初始化**以处理请求。

`cudaDeviceReset` 会销毁当前设备的主上下文。

若在主上下文被**销毁后调用** CUDA 运行时 API，系统将为该设备**创建新的主上下文**。

由于CUDA接口的全局状态会在主机程度**启动时初始化**，因此在主机程序启动期间或 main 函数执行后的终止阶段使用CUDA接口（隐式或显式）将导致未定义行为。

</aside>

## CUDA中的错误检查 Error Checking in CUDA

每个 CUDA API 都会返回一个**枚举类型** `cudaError_t` 的值。

最佳实践是**始终检查并管理**每个 CUDA API 调用的返回值。当没有错误时，返回的值为 `cudaSuccess`

```cpp
#define CUDA_CHECK(expr_to_check) do {    \
		cudaError_t result = expr_to_check;   \
		if(result != cudaSuccess)             \
		{                                     \
				fprintf(stderr,                   \
								"CUDA Runtime Error: %s:%i:%d = %s\n", \
								__FILE__,                 \
								__LINE__,                 \
								result,                   \
								cudaGetErrorString(result)); \ 
		}                                      \
} while(0)                               
```

<aside>
💡

do-while(0) 结构确保宏在**分支语句**中正确运行，__FILE__ 文件名宏 __LINE__ 当前行宏。

该宏使用 `cudaGetErrorString` API，该 API 返回描述特定 `cudaError_t` 值含义的**可读字符串**。

</aside>

### 错误状态 Error State

CUDA 运行时为**每个主机线程**维护一个 `cudaError_t` 状态。该值默认为 `cudaSuccess` ，并在发生错误时**被覆盖**。 

`cudaGetLastError` 返回当前错误状态后将其重置为 `cudaSuccess` 。或者， `cudaPeekLastError` 可在不重置的情况下返回错误状态。

使用三重尖括号语法启动内核**不会返回** `cudaError_t`。最佳实践是在**内核启动后立即检查错误状态**，以检测**内核启动中**的即时错误或**内核启动前**的异步错误。

若返回 `cudaSuccess` 值并不意味着内核已成功执行甚至开始执行，**仅表明传递给运行时的内核启动参数和执行配置未触发任何错误**，且错误状态并非内核启动前存在的历史错误或异步错误。

### 异步错误 Asynchronous Errors

CUDA 内核启动和许多运行时 API 均为**异步操作**。异步操作期间发生的错误，仅在下次检查错误状态时才会被报告。

<aside>
💡

如任何返回 `cudaError_t` 的 CUDA API 调用`cudaGetLastError` 或  `cudaPeekLastError`

</aside>

```cpp
vecAdd<<<blocks, threads>>>(devA, devB, devC, vectorLength);
// Check error state for kernel launch
CUDA_CHECK(cudaGetLastError());
// The CUDA_CHECK will report errors that occurred during excution of the kernel
CUDA_CHECK(cudaDeviceSynchronize()); 
```

`cudaEventQuery`和 `cudaStreamQuery`  函数可能返回的 `cudaError_t` 值 cudaErrorNotReady，不被视为错误，也不会被 `cudaPeekLastError`或 `cudaGetLastError` 函数报告，如下代码：

```cpp
cudaError_t err = cudaStreamQuery(stream);

if (err == cudaErrorNotReady) {
    // 这不是错误！只是表示"流还未准备好"
    printf("Stream busy, try again later\n");
}

// 但 cudaErrorNotReady 不会被错误处理函数捕获
cudaError_t lastErr = cudaGetLastError();  
```

### CUDA_LOG_FILE

另一种识别 CUDA 错误的有效方法是使用 `CUDA_LOG_FILE` 环境变量。**设置此环境变量**后，CUDA 驱动程序会将遇到的错误信息**写入该环境变量指定路径的文件中**。如下：

```bash
env CUDA_LOG_FILE=cudaLog.txt ./errlog
```

将 `CUDA_LOG_FILE` 设置为 `stdout` 或 `stderr` 将分别输出至标准输出与标准错误流。

## 设备与主机函数 Device and Host Functions

通常内核函数由`__global__` 修饰符标识并通过主机端调用启动，但也可通过**动态并行机制**在**内核函数内部启动**另一个内核函数。

限定符 `__device__` 表示函数应编译为 GPU 可执行代码，且能被其他 `__device__`或`__global__`函数调用。

## **变量说明符 Variable Specifiers**

CUDA 说明符可用于**静态变量声明**，以控制其**存储位置**。

- `__device__` 指定变量存储在全局内存中
- `__constant__` 指定变量存储在常量内存中
- `__managed__` 指定变量存储为统一内存
- `__shared__` 指定变量存储在共享内存中

当变量在 `__device__` 或 `__global__` 函数**内部声明**且**不带限定符**时，将尽可能分配至**寄存器**，必要时分配至**局部内存**。

### 检测设备编译

函数（包括类成员函数、仿函数和 Lambda 表达式）可同时声明为 `__device__` 与 `__host__`

当函数被指定为 `__host__ __device__` 时，编译器将为其**同时生成 GPU 与 CPU 代码**。检查 `__CUDA_ARCH__` 是否被定义是最常用的实现方式。

## **线程块集群 Thread Block Cluster**

自**计算能力 9.0 版本**起，CUDA 编程模型引入了一个名为**线程块集群**的可选层次结构，该集群由线程块组成。**【4060ti为8.9。计算能力9.0则为H100以上的专业计算卡或RTX50系列显卡】**

![image.png](2%201%20CUDA%20C++/image.png)

与线程块类似，簇也组织成一维、二维或三维的线程块簇网格。

集群中的线程块数量可由用户定义，在 CUDA 中可移植集群大小**最多支持 8 个线程块**。

- 集群中的所有线程块均保证**协同调度**，在单个 GPU 处理集群（GPC）上同步执行，并支持集群内的线程块通过**协作组 API** `cluster.sync()` 实现硬件支持的同步操作。
- 集群组还提供成员函数，可分别通过 `num_threads()` 和 `num_blocks()` API **查询线程数或块数的集群组规模**。
- 线程或块在集群组中的**秩**可分别通过 `dim_threads()` 和 `dim_blocks()` API 进行查询。
- 属于集群的线程块可访问**分布式共享内存**，即集群内**所有线程块的共享内存总和**。集群中的线程块能够对分布式共享内存中的**任意地址进行读取、写入及原子操作**。

### **使用三重尖括号表示法启动集群**

线程块集群可以通过两种方式在核函数中启用：一种是通过**编译时核函数属性**使用 `__cluster_dims__(X,Y,Z)` ，另一种是通过 **CUDA 核函数**启动 API `cudaLaunchKernelEx` 。

```cpp
// Kernel definition
// Compile time cluster size 2 in X-dimension and 1 in Y and Z dimension
__global__ void __cluster_dims__(2, 1, 1) cluster_kernel(float* input, float* output)
{
		
}

int main()
{
		float *input, *output;
		// Kernel invocation with compile time cluster size
		dim3 threadsPerBlock(16, 16);
		dim3 numBlocks(N / threadPerBlock.x, N / threadPerBlock.y);
		
		// The grid dimension is not affected by cluster launch, and is still enumerated
		// using number of blocks
		// The grid dimension must be a multiple of cluster size.
		cluster_kernel<<<numBlocks, threadsPerBlock>>>(input, output);
}
```

上面的示例是用编译时核函数属性启动集群，集群大小在**编译时固定**，启动时无法修改集群规模。
