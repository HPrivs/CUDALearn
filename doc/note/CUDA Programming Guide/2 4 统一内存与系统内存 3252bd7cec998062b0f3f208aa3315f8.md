# 2.4 统一内存与系统内存

异构系统中存在多个可存储数据的物理存储器。当数据**驻留在访问它的处理器所属内存**中时，性能达到最优。

CUDA 提供了用于显式管理内存位置的 API，但这种方式可能较为繁琐且使软件设计复杂化。

CUDA 提供了专门用于简化不同物理存储器间数据分配、布局和迁移的功能与特性。

本章将介绍**以上特性**并说明**如何判断适用哪种统一内存模**式，以及统一内存的各项功能在每种模式下的行为表现。

本章将定义和解释以下概念：

- **统一虚拟地址空间（Unified Virtual Address Space）** ：CPU 内存与每个 GPU 内存在单一虚拟地址空间中拥有各自独立的地址范围。
- **统一内存（Unified Memory）：**一项CUDA功能，支持可在CPU和GPU之间自动迁移的托管内存
- **页锁定主机内存**：系统内存中不可换页的部分，为某些 CUDA 操作所必需

此外，以下在讨论统一内存和系统内存时使用的术语也在此处引入：

- **异构管理内存（Heterogeneous Managed Memory, HMM）**：Linux 内核的一项功能，通过在操作系统进程中为完整统一内存提供软件一致性支持
- **地址转换服务（Address Translation Service, ATS）：**一项硬件功能，在 GPU 通过 NVLink 芯片间（C2C）互连与 CPU 连接时可用，为完整统一内存提供硬件一致性支持

# **统一虚拟地址空间**

在**单个操作系统进程**内，系统**所有主机内存**及**所有 GPU 内存**共享**统一的虚拟地址空间**。

无论是通过 **CUDA API**（例如 `cudaMalloc` 、 `cudaMallocHost` ）还是**系统分配 API(**例如 `new`  、 `malloc` 、 `mmap` )进行内存分配，**这一原则均成立**。

CPU 与每个 GPU 在**统一虚拟地址空间**内**拥有各自独立的地址范围**。

这意味着：

- 任何内存的位置（无论主机内存，**GPU全局内存【注意：不包括SM中的寄存器文件、共享内存、L1缓存】**）都可以通过使用 `cudaPointerGetAttributes()` 从指针值确定
- `cudaMemcpy*()` 的 `cudaMemcpyKind` 参数可以设置为 `cudaMemcpyDefault` ，以**根据指针自动确定**复制类型

# 统一内存 Unified Memory

统一内存是 CUDA 的一种内存特性，它允许**从 CPU 或 GPU 上运行的代码访问**称为**托管内存(Managed Memory)**的内存分配。**【统一虚拟地址空间是实现这一特性的必要不充分条件，迁移是另外的部分】**统一内存适用于所有 CUDA 支持的系统。

在某些系统上，托管内存必须显式分配。托管内存可以**通过几种不同的方式**在 CUDA 中显式分配：

- `cudaMallocManaged`
- 使用通过 `allocType` 设置为 `cudaMemAllocationTypeManaged` 创建的池的 CUDA API `cudaMallocFromPoolAsync`
- 具有 `__managed__` 说明符的全局变量

在支持 **HMM 或 ATS 的系统上**，无论分配方式如何，所有系统内存均为**隐式托管内存**。无需进行特殊的内存分配操作。**【补充：HMM是一个核心的软件机制，允许GPU和CPU访问完整统一虚拟地址空间，ATS是一个硬件特性，依赖于PCIe总线协议，允许GPU直接使用CPU的页表】**

## **统一内存范式 Unified Memory Paradigms**

统一内存的具体特性与行为会因操作系统、Linux 内核版本、GPU 硬件以及 GPU-CPU 互连架构而异。

可通过使用 `cudaDeviceGetAttribute` 查询若干属性来确定可用的统一内存形式**【亦如下图，按顺序查询】**：

- `cudaDevAttrConcurrentManagedAccess` - 1 表示完全支持统一内存，0 表示支持有限
- `cudaDevAttrPageableMemoryAccess` - 1 表示全部**主机内存**均为完全支持的统一内存，0 表示仅显式分配的托管内存是完全支持的统一内存
- `cudaDevAttrPageableMemoryAccessUsesHostPageTables` - 指示 CPU/GPU 一致性机制：1 为硬件实现，0 为软件实现**【注：此处应该指的分别是HMM和ATS的实现方式】**

![**所有**当前 GPU **均采用统一虚拟地址空间**并**具备统一内存功能（cudaDevAttrManagedMemory始终为1）**。](2%204%20%E7%BB%9F%E4%B8%80%E5%86%85%E5%AD%98%E4%B8%8E%E7%B3%BB%E7%BB%9F%E5%86%85%E5%AD%98/image.png)

**所有**当前 GPU **均采用统一虚拟地址空间**并**具备统一内存功能（cudaDevAttrManagedMemory始终为1）**。

统一内存操作有**四种模式**：

- 完全支持显式托管内存分配
- 完全支持具有软件一致性的所有分配
- 完全支持具有硬件一致性的所有分配
- 有限的统一内存支持

### 统一内存范式：代码实例

```cpp
void queryDevices()
{
		int numDevices = 0;
		cudaGetDeviceCount(&numDevices);
		for(int i = 0; i < numDevices; i++)
		{
				cudaSetDevice(i);
				cudaInitDevice(0, 0, 0);
				
				int deviceId = i;
				
				int concurrentManagedAccess = -1;
				cudaDeviceGetAttribute(&concurrentManagedAccess, cudaDevAtitrConcurrentManagedAccess, deviceId);
				int pageableMemoryAccess = -1;
				cudaDeviceGetAttribute(&pageableMemoryAccess, cudaDevAttrPagableMemoryAccess, deviceId);
				int pageableMemoryAccessUsesHostPageTables = -1
				cudaDeviceGetAttribute(&pageableMemoryAccessUsesHostPageTables, cudaDevAttrPagableMemoryAccessUsesHostPageTables, deviceId);
				
				printf("Device %d has ", deviceId);
				if(concurrentManagedAccess){
						if(pageableMemoryAccess){
								printf("full unified memory support ");
								if(pageableMemoryAccessUsesHostPageTables)
								{printf("with hardware coherency.\n");}
								else
								{printf("with software coherency.\n")}
						}
						else
						{printf("full unified memory support for CUDA-made managed allocations\n");}
				}
				else
				{printf("limited unified memory support: Windows, WSL, or Tegra\n");}
		}
}
```

## 完整的统一内存功能支持

大多数 Linux 系统都具备完整的统一内存支持。

如果设备属性 `cudaDevAttrPageableMemoryAccess` 为 1，那么**所有的系统内存**，无论是通过 CUDA API 还是系统 API 分配的，都将作为具有完整功能支持的统一内存运行。包括使用 `mmap` 创建的**文件支持的内存分配**。**【理解：说明GPU可访问CPU的可分页内存】**

如果 `cudaDevAttrPageableMemoryAccess` 为 0，则只有由 **CUDA 分配为托管内存的内存**才表现为统一内存。使用**系统 API 分配**的内存**不受管理**，且不一定能从 GPU 内核访问。

对于具有完整支持的统一内存分配：

- 托管内存通常**首次分配在被访问的处理器内存空间。【理解：cudaMallocManaged仅在虚拟地址空间“占坑”，当CPU或GPU首次读写时，驱动程序在对应侧建立物理页映射表】**
- 当托管内存被**当前驻留处理器之外**的处理器使用时，通常会**发生迁移**
- 托管内存**按内存页**（软件一致性）或**缓存行**（硬件一致性）的粒度**进行迁移或访问【理解：内存页迁移迁移适合大块连续访问，缓存行访问适合随机微量访问，避免浪费】**
- **允许超额分配**：应用程序分配的托管内存可能超过 GPU 实际可用的物理内存容量**【当托管内存容量超过GPU物理显存容量，驱动程序启用LRU(Least Recently Used)将不常用页面调到主机内存】**

**分配（Allocation）和迁移（Migration）行为**可能偏离上述描述，程序员可通过**提示和预取（hints and prefetches）**影响这一行为。

## **支持硬件一致性的完整统一内存**

当使用 NVIDIA CPU 且 CPU 与 GPU 之间通过 **NVLink 芯片到芯片（C2C）**互连时，可提供**地址转换服务（Address Translation Service）**。

通过 ATS，除了对所有主机分配提供完整的统一内存支持外，还具备以下特性：

- GPU 分配可从 CPU 访问（ `cudaDevAttrDirectManagedMemAccessFromHost` 将为 1）
- CPU 与 GPU 之间的链路支持**原生原子操作**（ `cudaDevAttrHostNativeAtomicSupported` 将为 1）
- 相比软件一致性，硬件对一致性的支持能够提高性能

<aside>
💡

**ATS 提供了 HMM 的所有功能**。当 **ATS 可用时，HMM 会自动禁用**。

</aside>

## **具有软件一致性的完全统一内存**

**异构内存管理（HMM）**是 **Linux 操作系统**（需满足**相应内核版本**）提供的一项功能，它实现了基于软件一致性的完全统一内存支持。

异构内存管理为通过 **PCIe 连接的 GPU** 带来了 ATS所提供的**一些功能和便利**。

当 HMM 可用时，系统支持完全统一内存，并且**所有系统分配的内存**都隐式成为**统一内存**。

## **有限的统一内存支持**

在 **Windows（包括WSL）**以及**部分 Tegra 系统**上，仅提供有限的统一内存功能子集。

在这些系统中，虽然托管内存可用，但 CPU 与 GPU 之间的**迁移行为存在差异：**

- 托管内存**首先**被分配在 **CPU 的物理内存**中
- 托管内存的**迁移粒度大于虚拟内存页的粒度**
- 当 GPU 开始执行时，托管内存被**迁移到 GPU**
- 在 GPU 处于活动状态时，CPU 不得访问托管内存
- 当 GPU 同步时，托管内存被迁移回 CPU**【注：不支持CPU和GPU同时访问（如上点所述），同步意味着GPU任务结束，驱动会在“同步点”将内存搬回CPU侧】**
- **不允许** GPU 内存**超额分配**
- 只有由 **CUDA 显式分配**为托管内存的内存才是统一的

## **内存建议与预取 Memory Advise and Prefetch**

程序员可以向管理统一内存的 NVIDIA 驱动程序**提供提示**，以帮助其最大化应用程序性能。

`cudaMemAdvise` 允许程序员**指定内存分配属性**，这些属性会影响内存的**放置位置**，以及在从其他设备访问时**是否迁移内存**。

`cudaMemPrefetchAsync` 允许程序员建议启动**特定内存分配至不同位置的异步迁移**。一种常见用法是在内核启动前，**提前开始传输内核将使用的数据**。这使得数据拷贝可以在其他 GPU 内核执行的同时进行。

**【理解：cudaMemAdvise主要是声明规则，建议内存的存放位置和迁移内存规则。而cudaMemPrefetchAsync则是立刻发起异步迁移任务】**

# **页锁定主机内存 Page-Locked Host Memory**

`cudaMallocHost` 在 CPU 上分配**页锁定内存**（也称为**固定内存**）。

通过**传统分配机制（如 `malloc` 、 `new` 或 `mmap` ）**进行的主机分配**不具备页锁定属性**，这意味着它们可能被操作系统交换到磁盘或物理上**重新定位**。

CPU 与 GPU 之间的**异步拷贝**需要页锁定主机内存。

页锁定内存可以**映射到 GPU**，以便 GPU 内核直接访问。

---

CUDA 运行时提供 API 来分配页锁定主机内存或对现有分配进行页锁定：

- `cudaMallocHost` 分配页锁定主机内存
- `cudaHostAlloc` 默认行为与 `cudaMallocHost` 相同，但也可通过标志位指定其他内存参数
- `cudaFreeHost` 释放通过 `cudaMallocHost` 或 `cudaHostAlloc` 分配的内存
- `cudaHostRegister` 对 CUDA API 外部分配的内存区域（例如通过 `malloc` 或 `mmap` 分配）进行页锁定**【理解：对现有分配进行页锁定，如第三方库或开发者控制之外的代码分配的主机内存】**

<aside>
💡

系统中**所有 GPU** 均可将页锁定主机内存用于**异步拷贝**和**映射内存**操作。

</aside>

## 映射内存 Mapped Memory

在支持统一地址管理或地址转换服务的系统中，所有主机内存均可**通过主机指针直接由 GPU 访问**。

当不具备地址转换服务或统一地址管理功能时，可以通过将**内存映射到 GPU 内存空间的方式**使主机分配的内存对 GPU 可访问。

**《映射内存始终是页锁定的。》**

**【理解：系统中所有GPU均可访问页锁定主机内存的原因是页锁定内存是固定物理地址，通过映射内存可以实现所有GPU访问页锁定内存。在没有相应的服务情况下，GPU不具备访问非页锁定内存的能力。】**

```cpp
__global__ void copyKernel(float* a, float* b)
{
        int idx = threadIdx.x + blockDim.x * blockIdx.x;
        a[idx] = b[idx];
}
```

<aside>
💡

在内核中访问映射内存需要通过 **CPU-GPU 互连、PCIe 或 NVLink C2C** 进行数据传输。

与访问设备内存相比，这些操作具有**更高的延迟和更低的带宽**。

对于内核的大部分内存需求而言，**映射内存不应被视为**统一内存或显式内存管理的**高性能替代方案**。

</aside>

### **cudaMallocHost 与 cudaHostAlloc**

通过 `cudaHostMalloc` 或 `cudaHostAlloc` 分配的主机内存会**自动映射**。这些 API 返回的**指针**可直接在内核代码中用于**访问主机端内存**。主机内存通过 **CPU-GPU 互连**进行访问。

### cudaMallocHost代码示例

```cpp
void usingMallocHost()
{
		float* a = nullptr;
		float* b = nullptr;
		
		CUDA_CHECK(cudaMallocHost(&a, vLen * sizeof(float)));
		CUDA_CHECK(cudaMallocHost(&b, vLen * sizeof(float)));
		
		initVector(b, vLen);
		memset(a, 0, vLen * sizeof(float));
		
		int threads = 256;
		int blocks = vLen / threads;
		copykernel<<<blocks, threads>>>(a, b);
		CUDA_CHECK(cudaGetLastError());
		CUDA_CHECK(cudaDeviceSynchronize());
		
		printf("Using cudaMallocHost: ");
		checkAnswer(a, b);
}

```

### cudaAllocHost代码示例

```cpp
void usingcudaAllocHost()

		float* b = nullptr;

		CUDA_CHECK(cudaDeviceSynchronize());
		
		printf("Using cudaAllocHost: ");
		checkAnswer(a, b);
}

```

### cudaHostRegister

当 ATS 与 HMM 不可用时，仍可通过 `cudaHostRegister` 将**系统分配器创建的内存**映射为可直接从 **GPU 内核访问的形式**。

但此类内存**无法通过主机指针**在内核中访问。必须使用 `cudaHostGetDevicePointer()` **获取设备内存区域的指针**，并在内核代码中使用该指针进行访问。

```cpp
void usingRegister()
{
		float* a = nullptr;
		float* b = nullptr;
		
		a = (float*)malloc(vLen * sizeof(float));
		b = (float*)malloc(vLen * sizeof(float));
		
		CUDA_CHECK(cudaHostRegister(a, vLen * sizeof(float), 0));
		CUDA_CHECK(cudaHostRegister(b, vLen * sizeof(float), 0));
		
		CUDA_CHECK(cudaHostGetDevicePointer((void **)&devA, (void *)a, 0));
		CUDA_CHECK(cudaHostGetDevicePointer((void **)&devB, (void *)b, 0));
		
		initVector(b, vLen);
		memset(a, 0, vLen * sizeof(float));
		
		int threads = 256;
		int blocks = vLen / threads;
		copykernel<<<blocks, threads>>>(devA, devB);
		CUDA_CHECK(cudaGetLastError());
		CUDA_CHECK(cudaDeviceSynchronize());
		
		printf("Using cudaHostRegister: ");
		checkAnswer(a, b);
}
```

### **统一内存与映射内存对比**

映射内存使得 CPU 内存可从 GPU 访问，**但无法保证所有访问类型（例如原子操作**）在所有系统上都受支持。

映射内存**始终驻留在 CPU 内存中**，这意味着所有 GPU 访问都必须通过 CPU 与 GPU 之间的连接（PCIe 或 NVLink）进行。链路访问的延迟显著高于访问 GPU 内存，且总可用带宽较低**【注：映射内存也称零拷贝内存】**

统一内存则保证支持**所有访问类型**。

统一内存通常**会迁移至访问它的处理器的物理内存中**。首次迁移后，**内核对相同内存页或缓存行的重复访问可充分利用 GPU 内存带宽**。

# 总结

- 在支持异构内存管理（HMM）或地址转换服务（ATS）的 Linux 平台上，所有系统分配的内存均为托管内存
- 在不支持 HMM 或 ATS 的 Linux 平台、Tegra 处理器以及所有 Windows 平台上，托管内存必须通过 CUDA 接口进行分配
- 在 Windows 和 Tegra 处理器上，统一内存存在限制
- 在配备 ATS 的 NVLINK C2C 互联系统中，使用 `cudaMalloc` 分配的设备内存可直接从 CPU 或其他 GPU 访问
